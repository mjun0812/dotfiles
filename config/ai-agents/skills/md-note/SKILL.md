# md-note

## 目的

この Skill は、**現在の会話セッションで整理・調査した内容**を、
**日本語の Markdown 1 ファイル**にまとめて、**カレントディレクトリにファイルとして保存**するためのものです。

新規調査を主目的にはしません。**追加の検索や再調査は、ユーザーが明示した場合のみ**行ってください。

## 絶対ルール

ユーザーが明示的に上書きしない限り、必ず次を守ってください。

1. **Markdownの先頭に目次を置くこと。**
2. **再検索しなくてもMarkdownを読めば内容がわかる粒度で書くこと。**
3. **参考リンクがあれば入れること。**
4. **会話中ですでに確立した情報を優先してまとめること。**
5. **事実・推論・未確認事項を混同しないこと。**
6. **不確実性は曖昧にせず明記すること。**
7. **1つのMarkdownファイルにまとめること。**

## 保存ファイル名ルール

出力するMarkdownのファイル名は、必ず次の形式にしてください。

```text
YYYYMMDD_<topic>.md
```

例:

- `20260410_market-overview.md`
- `20260410_ai-regulation-summary.md`
- `20260410_research-summary.md`

### 命名規則

- `YYYYMMDD` は作成日。
- `<topic>` は内容を短く表す識別子。
- `<topic>` は原則として **英小文字・数字・ハイフン**を使う。
- 適切なトピック名が付けにくい場合は **`research-summary`** を使う。
- 空白は使わない。
- 拡張子は必ず `.md`。

## 作業手順

### 1. 対象範囲を確定する

ユーザーの依頼から、まとめる対象を推定してください。

- 「この会話」「ここまでの調査」なら、その範囲全体を対象にする。
- 特定テーマが指定されていれば、そのテーマだけを対象にする。
- 範囲が曖昧でも、合理的な前提を置いてまとめ、文書内に前提を書いてよい。

### 2. 追加調査を勝手にしない

この Skill の目的は**まとめること**です。

以下を優先して利用してください。

- この会話で確立した結論
- 途中で整理された比較・論点
- すでに提示済みの数字・日付・条件
- 会話中に参照済みのリンク
- 未解決の論点や保留事項

ユーザーが「追加で調べてからまとめて」と明示した場合のみ、追加調査を行ってください。

### 3. Markdown の構成を組み立てる

テンプレートは `references/OUTPUT_TEMPLATE.md` を参照してください。

必要に応じて節は増減して構いませんが、次は維持してください。

- タイトル
- 先頭の目次
- 概要または要約
- 結論
- 詳細
- 参考リンク

### 4. 自己完結する粒度で書く

読み手は**元の会話を見ていない**前提で書いてください。

そのため、本文には以下を含めてください。

- 何についての文書か
- なぜその調査が必要だったか
- 比較対象や選択肢の名前
- 数値、日付、条件、制約
- 結論に至る理由
- どこまで確認済みか

「いくつか候補を比較した」のような曖昧表現は避け、候補名を書けるなら書いてください。

### 5. 参考リンクの扱い

参考リンクは必ず含めてください。

- 末尾に `## 参考リンク` セクションを置く。
- 本文中でも理解に有用なら Markdown リンクを入れる。
- 会話中の引用や出典が URL 化できるなら、読みやすいリンクにする。
- URL が不明な出典は、名称だけでも残してよいが、未確認と明記する。

### 6. 事実と推論を分ける

- **事実**: 会話内で確認済み、もしくは参照済みソースに基づく内容
- **推論**: 複数情報からの整理・要約・解釈
- **未確認事項**: 追加確認が必要な内容

必要なら本文で明示的にラベルを付けてください。

## 保存時の実務ルール

次の流れで処理してください。

1. 完成した Markdown を生成する。
2. 適切な `<topic>` を決める。
3. `scripts/write_markdown.py` を使ってカレントディレクトリに保存する。
4. 保存後、ユーザーには簡潔に以下を返す。
   - 何をまとめたか
   - 保存先ファイル名
   - 必要なら対象範囲の前提

## 品質チェック

最終化前に必ず確認してください。

- 目次が先頭にあるか
- Markdown だけで理解できるか
- 結論が前半にあるか
- 数字・日付・条件が十分に入っているか
- 参考リンクがあるか
- 未確認事項が明記されているか
- ファイル名が `YYYYMMDD_*.md` 形式か
- 概要は 3 行以内に収まっているか
- 全体の文字数が 500〜3000 字程度を目安にしているか（内容に応じて超過は許容）
- `scripts/write_markdown.py` を使って保存したか

## スクリプト利用（必須）

ファイル保存には**必ず** `scripts/write_markdown.py` を使用してください。
Write ツールで直接書くことは禁止です。

スクリプトは以下を自動で行います。

- 日付 (YYYYMMDD) の自動計算（システムのローカルタイムゾーンを使用）
- topic slug の正規化（記号除去・小文字化）
- 同名ファイルの上書き防止（デフォルトで上書きを拒否する）

### 使い方

```bash
python scripts/write_markdown.py --stdin --slug research-summary
```

このコマンドは、標準入力から受け取った Markdown を、カレントディレクトリに
`YYYYMMDD_research-summary.md` として保存します。

同名ファイルが既に存在する場合はエラーになります。上書きするには `--force` を付けてください。

## 例

| ユーザーの依頼                         | slug 例                 | ポイント                       |
| -------------------------------------- | ----------------------- | ------------------------------ |
| 「競合比較を Markdown にまとめて」     | `competitor-comparison` | 対象テーマだけ抽出             |
| 「ここまでの調査を引き継ぎメモにして」 | `research-summary`      | 追加調査しない・未解決事項明記 |
| 「この規制調査だけ md にして」         | `regulation-summary`    | 規制調査のみ抽出               |

いずれも `scripts/write_markdown.py --stdin --slug <slug>` で保存する。
