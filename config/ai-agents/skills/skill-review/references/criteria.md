# 評価基準の詳細

skill-reviewの観点1〜12の判定基準。各観点は「合格の条件」を満たせば合格、
1つでも「要改善のシグナル」に該当すれば要改善とする。

## 1. 目標と境界

合格の条件:

- skillの目的(何の問題を解決するか)が冒頭付近で述べられている
- やること・やらないことの境界が読み取れる(明示のセクションが無くても、手順から一意に読み取れれば良い)

要改善のシグナル:

- 目的の記述が無く、手順の羅列から推測するしかない
- スコープ外の動作(例: コミットするのか、pushまでするのか)が不明

良い例:

```text
## 目的
ドキュメントと実装の乖離を検出して更新するSkillです。

## 原則
- コミットしない: このSkillは更新までで終わる。
```

## 2. description品質

frontmatterの`description`はskillを発動するかどうかの判断材料になる。
本文を読まなくても発動判断ができることが基準。

合格の条件:

- 何をするskillかが1文目で分かる
- 発動すべき典型的な依頼フレーズが含まれる(例: 「〜して」と依頼したら使うこと)
- 誤発動しやすい隣接ケースがあるなら「〜の場合は使わない」が書かれている

要改善のシグナル:

- 機能説明のみでトリガー条件が無い
- 他のskillのdescriptionと発動条件が重複・衝突している

悪い例:

```yaml
description: PRを作成するSkill。
```

良い例:

```yaml
description: >-
  現在のbranchからPull Requestを作成するSkill。
  ユーザーが「PR作って」「pull request作成して」と依頼したら使うこと。
  レビューまで求められた場合はこのskillではなくgithub-pr-create-self-reviewを使う。
```

## 3. コマンド例の量

AIはコマンド自体の使い方を知っているため、例は情報を追加する場合にだけ価値がある。

合格の条件:

- コマンド例が次のいずれかを明確にする場合にだけ存在する
  - 入力・出力の形式
  - 操作の順序
  - 境界ケース

要改善のシグナル:

- `git status`, `ls` など自明なコマンドの例が並んでいる
- 同じ趣旨の例が網羅性のために複数ある

## 4. 完了条件

合格の条件:

- 何をもって成功とするかが定義されている(例: 「レポートを出力して終了」「テストが通ること」)
- 最終的にユーザーへ何を提示するかが分かる

要改善のシグナル:

- 手順が途中で終わっていて、成果物や終了状態が不明

## 5. 早期リターン

前提が満たされない状況で作業を続けるとトークンを浪費する。

合格の条件:

- 前提条件(対象ファイルの存在、リポジトリであること、必要コマンドの存在など)を最初に確認し、満たさなければ即終了する手順がある

要改善のシグナル:

- 前提が壊れているケースへの言及が無く、失敗したまま手順を続ける余地がある
- 前提チェックが手順の後半にある

該当なしの条件:

- 前提となる外部状態が実質的に存在しないskill(純粋な文章作成など)

## 6. 自己完結性

合格の条件:

- 他のskillへ処理を委譲していない
- 例外: 複数skillの連結が目的のオーケストレーションskillは、連結先の列挙のみ許容

要改善のシグナル:

- 手順の途中で「詳細は〜skillを参照」のように他skillの本文へ依存している
- 依存skillが無いと単体で成立しない

## 7. 簡潔さ

合格の条件:

- SKILL.md本文が目安500行以下
- 各セクションが目的に対して必要な情報だけを含む

要改善のシグナル:

- 同じ指示の言い換え・繰り返しがある
- 起こり得ないケースへの防御的な指示が多い

## 8. 分割

合格の条件:

- 長い出力テンプレート・大量の例・詳細な基準表がreferences/等の別ファイルに分離され、本文からリンクされている
- 本文には「いつそのファイルを読むか」が書かれている

要改善のシグナル:

- 数十行を超えるテンプレートや例が本文に直接埋め込まれている

該当なしの条件:

- 本文が短く、分割する対象が存在しない

## 9. 可読性

合格の条件:

- 見出し構造が手順の流れと一致している
- 表・リスト・コードブロックが適切に使い分けられている
- 人間が読んで手順を再現できる

要改善のシグナル:

- 1セクションに無関係な話題が混在している
- 手順の番号や参照(「手順3で〜」)が実際の構成とズレている

## 10. dry-run

該当条件: ファイルの編集・作成・削除、または外部への書き込み(コミット、issue作成など)を伴うskillのみ。
読み取り専用skillは「該当なし」。

合格の条件:

- `--dry-run`等、変更を適用せずに差分・予定内容を提示するモードがある
- そのモードで書き込み系の操作を一切行わないことが明記されている

## 11. 鮮度

合格の条件:

- 本文が参照するファイルパス・スクリプト・付属ファイルが実在する
- 本文が前提とするコマンドが実在する(バージョン差異までは問わない)

要改善のシグナル:

- リンク先のreferencesファイルが無い
- 存在しないパスやリネーム済みのコマンドを参照している

判定方法: 本文からパス・コマンド名を抽出し、実在を確認する。
外部URLは到達確認まではせず、明らかに廃止されたものだけ指摘する。

## 12. モデル汎用性

skillはClaude以外のモデルや下位モデルでも使われる前提で書く。

合格の条件:

- 手順が特定モデルの固有機能(特定のtool名、特定製品のhookやSubAgentなど)に依存していない。
  依存する場合は、無い環境での代替手段か「無ければスキップ」が書かれている
- 手順が具体的で、推論力の低いモデルでも順に実行すれば再現できる
- 「賢く判断して」のような、モデルの能力に丸投げする指示が要所に無い。
  ただし、探索手段の選択など**手段の自由度**を残す記述は減点しない

要改善のシグナル:

- 特定モデル・特定エージェント製品でしか解釈できない指示がある
- 重要な分岐の判断基準が書かれておらず、モデルの推論任せになっている
